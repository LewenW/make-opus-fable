# 三臂客观评测:Opus 裸奔 vs Opus+套件 vs Fable 裸奔 — 2026-07-04/05

## 问题与方法

回答三个问题:①复杂/产品代码任务上装 skill 有没有用?②装了 skill 的 Opus 能否逼近 Fable?③差距弥合率 (B−A)/(C−A) 是多少?

- **三臂**:A = Opus 4.8 裸奔;B = Opus 4.8 + 套件(CLAUDE-core + long-horizon-protocol + verify-before-done 全文注入);C = Fable 5 裸奔。每臂 agent 有完整工具权(可写码、跑测试、迭代),独立工作目录。
- **评分完全客观**:每任务配隐藏测试(参赛臂永远看不到),按断言通过数打分——正确性维度,与文风无关,盲测轮辛苦消除的"文风混淆"在此结构性不存在。
- **评分器先自测**:每个任务先写参考解(应满分)+ 故意写坏的解(应显著掉分),评分器通过区分性检验才上场。自测抓出过 3 个评分器 bug(含一个会把非原子解误判为 0 分的崩溃传染缺陷)。

## 任务(6 个,难度递增)

## 这个"零区分"说明什么(核心结论)

1. **分钟级、自包含、规格可判定、允许自测的任务上,Fable-Opus 差距 = 0,skill 增益 = 0。** 这是 25 臂零反例的直接测量,与调研预测一致(差距随任务长度/开放度放大),现在是一手数据。执行反馈是均衡器:只要 agent 能跑自己的代码,规则再密(h1 的 28 条互锁)、文档再骗人(h2)、需求再漂移(h3),两个模型都能把自己修到满分。
2. **h3 的方法学教训**:5 份矛盾纪要一次性全部可见时,"合并"对两个模型都是易事。文献里 39% 的多轮跌幅发生在需求**跨轮次陆续到达、模型在中间就开始答**的场景——复现它需要真正的多轮交互 harness,一次性投喂复现不了。
3. **所以公开基准的大差距(FrontierCode Diamond 29.3 vs 13.4、Every 91 vs 63)全部来自小时级任务**——那个量级在本地评测框架内跑不动(单臂数小时 × 6 臂)。**"装 skill 的 Opus 离 Fable 有多近"在小时级任务上目前只能引用间接量化**:system card 图 8.2.A(SWE-bench Pro:Opus xhigh 68.6 vs Mythos low 75.0,差 6.4pp)、Confucius(强脚手架 ≈ 抬一个模型档位)、以及 skill 只能作用于其中"流程可靠性"分量的定性判断。

## 对使用方式的直接推论

- **短任务(分钟级)别开套件**——正确性没有增益,token 开销是净成本。skill 的 description 已按此收窄;CLAUDE-core 的纪律条款(诚实汇报、范围克制)例外,它们改的是行为质量不是正确率,盲测 15-2-0 的胜绩在那个维度。
- **套件的目标场景仍是长任务**(多轮、跨会话、无法廉价自测的),那里有文献与基准证据但本项目尚无一手测量。
- Fable 配额宝贵(07-07 后计费):**分钟级任务用 Opus 4.8 毫无损失**——这是本轮最省钱的结论。
