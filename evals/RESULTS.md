## 诊断与修复史(最有价值的部分)

**R1 崩盘根因**(6 败全部排雷满分,零实质失误):
1. 把验证过程演进交付物("Verified by running…"、"proven by direct execution"、写作任务先来一段元叙述)
2. 对着材料齐全的文本题写"我无法实际执行"的免责块
3. 共享 scratchpad 串场(两臂互捡对方的测试文件写进答案)——harness 缺陷,对越肯真跑验证的臂伤害越大
→ **结构性根因:v1 把"每条声明对照 tool result 审计"放在第一节,模型理解成"要展示证据";"findings 不是 narration"藏在第三节。primacy 效应反噬。**(与 iwoszapar v1 四连败同病)

**v2 手术**:Rule 0 提到最顶("读者只看到 findings")、证据审计改为内部动作、新增"最便宜的充分检查"(材料在眼前时读它就是检查)、工作区噪音禁入;core 加"无前言"。→ 纯叙述类败仗全翻。

**v3 手术**(针对 R2 残留):禁语清单(任何以"我检查了"为主语的句子)、每个发现只说一次、caveat 预算(至多一条一行且须改变读者行动)、宣称矛盾必须引用双方条款(修复 wallet 题把"永不返回负 vs 抛异常"误判为自相矛盾的失误)、"返回新对象"合同必查引用别名。→ R3 4/6。

## 两场遗留败仗(如实保留)

1. **live-state-truth-T3**(三轮皆输,逐轮变窄):双方每轮都排雷满分,套件臂始终多出一点重量(第三轮只剩一段收尾 hedge + 结论复述)。此题型(运维两事实速查)极度奖励压缩,是该 skill 的最弱适用场景——iwoszapar 的同型 skill 在此也只拿 tie。**结论:无增益区,不再迭代。**
2. **heldout-cli-exit-codes**:套件臂抓满 3 雷 + bonus,败于一句附加推理自相矛盾(正确性缺陷压过对手更重的叙述扣分)。评审明示"单看简洁度套件本赢"。**结论:单次推理方差,非 skill 缺陷。**

## 诚实的限制

- n=1/题/轮,LLM 盲评(从不判 tie,比 iwoszapar 的评审更果断→边缘题变成硬输赢);三个 held-out 任务的出题人=改 skill 的人(与 iwoszapar 相同的"作者重合"局限);R1 期间会话模型被切换过,评审模型在 fable/opus 间不完全一致。
- **最重要的空白:这 17 题全部是短的单轮任务**——只验证了"短任务纪律层"。套件的核心目标(长视野多步任务,eval-plan 的 7-12 号种子)需要多轮 harness 才能测,**尚未验证**。长任务组才是 Fable-Opus 差距真正所在。
- token 开销未精确分账;skill 正文 ~1.2-1.5k token,量级与先例的 ~7% 一致。

## 教训(可复用于任何 rigor skill)

1. 行为规则的**位置**就是权重:必须让"输出纪律"压过"过程纪律"(Rule 0 置顶)。
2. 验证类 skill 的默认失败模式是把模型变成"表演验证的模型"——先例和本项目 v1 双双踩中,这条规律现在有两个独立数据点。
3. 修 A 会挤出 B(wallet 题:强调找矛盾 → 诱发误报矛盾),每刀都要 held-out 验证。
4. 评审 rubric 里"verbosity 不是 quality"这一句几乎决定一切边缘判罚——写 skill 时把它当第一性约束。
5. 无增益区要明写进 description 收窄触发,别让 skill 在自己的弱场景吃 token 还丢分。
